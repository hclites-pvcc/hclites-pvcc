Chapter 7: Software Engineering

TERM DEFINITION
integrated development environment systems that combine tools for developing software into an integrated package
Association for Computing Machinery
  • purpose of ACM: connects computing educators, researchers and professionals to collaborate within the field of computing
  • benefits of ACM membership: support in education, career development, professional networking
  • student ACM membership: provides resources specific to students in the field, including industry peers
Institute of Electrical and Electronics Engineers
  • purpose of IEEE: a technical professional organization dedicated to advancing technology through publications, conferences, technology standards, and professional/education activities
  • benefits of IEEE membership: Networking with other professionals, highlight your accomplishments, join discusions on technical interests, collaborate on projects, access to IEEE events and activities
  • student IEEE membership: Provides additional access to career resources, award program opportunities, networking, continuing education, discounts on products and services related to the field
Software Life Cycle: Development phase
  • Requirements analysis: the step in which the services to be provided by the proposed software are specified, as well as any conditions of those services, and how the world will interact with the software
  • Design: the plan for construction of the software, in which a solution to the problem being addressed is dveloped -- ends in a detailed description of how the structure can be converted into a functional program
  • Implementation: the actual writing of the program, creating data files, and developing databases
  • Testing: a phase in which developers seek confirmation that their software works as intending -- some developers say today that testing should not be a distinct phase of development, but rather incorporated into the other three phases
modular implementation a paradigm in which functions are implemented as basic modular constituents that results in a modular structure in which large tasks (or functions) are broken down into smaller ones
Unified Modeling Language (UML) a collection of tools developeed with OOP in mind to visually represent features of a program
use case an interaction between the system and its user
use case diagram a visual representation which shows how the user will interact with the software
Data relationships:
  • one-to-one: each data point corresponds to only ONE other data point
  • one-to-many: each data point of one entity type (ie. X) corresponds to many data points of the other entity type (ie. Y)
  • many-to-many: each data point corresponds to multiple other data points regardless of the entity type (ie. X or Y)
design pattern a pre-developed model for solving a recurring problem in software design
Software quality assurance (SQA) groups that oversee and enforce the quality standards held by a given software development organization
Software testing:
  • glass box testing: testing where the developer knows the interior structure of the software
  • black box testing: testing performed from the user's point of view where there is no knowledge of the interior structure of the software
  • beta testing: a type of black-box testing where a preliminary version of a software is given to a limited audience to see how the software performs when used as intended
  • alpha testing: similar to beta testing but performed at the developer's site
Documentation:
  • user documentation: a type of documentation where the features of the software and how to use them are explained
  • system documentation: a type of documentation where the software's internal composition is described for the purpose of future program maintenance by developers
  • technical documentation: a type of documentation where the installation and service (ie troubleshooting) procedures are described
Intellectual property law
  • software license: an expression of the legl terms of the software developer's rights
  • software disclaimer: an expression of the limitations of the developer's liability
  • best defenses against software liability claims: apply sound software engineering principles during the development process, use care that is compatible with the use of the software, and document these efforts with well kept records
